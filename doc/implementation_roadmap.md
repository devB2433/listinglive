# 开发计划建议（分模块 · 分步骤）

**项目**: ListingLive / Vivid · 房地产图生视频服务  
**依据**: rfc.txt、requirements.md、technical_design.md、development_plan.md  
**原则**: 按模块拆分、按依赖分步、每步可验收

---

## 一、模块划分与依赖关系

```
┌─────────────────────────────────────────────────────────────────────────┐
│  前端 · 通知 · 管理后台（四模块之后可再按同样粒度拆分）                    │
└─────────────────────────────────────────────────────────────────────────┘
                                    │
│  业务模块（M0 完成后）  用户管理 → 套餐管理 → 短视频任务 → 长视频任务    │
│                                    │                                     │
│  ┌─────────────────────────────────┼─────────────────────────────────┐   │
│  │ 基础架构 M0                     ▼                                 │   │
│  │  项目骨架 · 配置 · 数据库 · 存储(本地→S3) · Celery · Docker        │   │
│  └──────────────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────────────┘
```

**模块清单**（按建议开发顺序）：

| 序号 | 模块 | 说明 | 主要产出 |
|------|------|------|----------|
| **M0** | **基础架构** | 项目骨架、配置、数据库、存储、任务队列 | 可运行的后端 + 本地/容器环境 |
| **一** | **用户管理** | 注册/登录、用户模型、JWT、测试账号、用户信息 | 用户可注册登录、可查 profile |
| **二** | **套餐管理** | 套餐与配额包配置、订阅、Stripe、配额校验 | 用户可订阅、扣配额、查配额 |
| **三** | **短视频任务** | 视频服务抽象、场景模板、单图→单视频任务、查询与下载 | 单图生成单视频 E2E |
| **四** | **长视频任务** | 多图→多片段→合并、转场与水印、任务与下载 | 多图合并成长视频 E2E |

**约定**：M0 完成后，按 **用户管理 → 套餐管理 → 短视频任务 → 长视频任务** 的粒度模块化开发；长视频任务复用短视频任务中的视频服务抽象、任务表与下载 API。通知、管理后台、前端可在四模块之后另行拆分。

## 二、分模块分步骤计划

### 模块 M0：基础架构

**目标**：搭好后端骨架，支持配置驱动、数据库、对象存储（短期本地、抽象兼容 S3）、异步任务，无业务逻辑。

| 步骤 | 内容 | 验收标准 |
|------|------|----------|
| M0.1 | 项目初始化 | FastAPI 项目、目录结构（api / core / models / services / tasks）、pyproject.toml 或 requirements.txt |
| M0.2 | 配置管理 | 环境变量 + 配置文件（.env.example），区分 dev/test/prod，无硬编码敏感信息 |
| M0.3 | 数据库 | PostgreSQL + SQLAlchemy 2.0 模型定义、连接池、Alembic 迁移脚本（先建 users 等核心表） |
| M0.4 | Redis | Redis 连接、用于 Celery broker + 后续缓存 |
| M0.5 | Celery | Celery 接入 Redis、一个 hello 任务可被触发并执行 |
| M0.6 | 对象存储抽象 | **短期**：服务器本地目录存储；**接口**：统一抽象（上传/下载/获取访问 URL），业务只依赖接口；**后续**：通过实现同一接口切换为 S3，无需改业务代码 |
| M0.7 | Docker Compose | 本地一键起 PG + Redis + 后端 API + Worker，文档说明如何启动 |

**建议周期**：约 1 周（若已有模板可缩短）。

### 模块一：用户管理

**目标**：用户使用**用户名+密码+邮箱**注册，系统发**邮件验证码**防欺诈；登录使用**用户名或邮箱+密码**，接口返回 JWT。提供初始测试账号 root/root 便于测试。

| 步骤 | 内容 | 验收标准 |
|------|------|----------|
| 1.1 | 用户模型与表 | users 表（id, username 唯一, password_hash, email, email_verified, status 等），迁移脚本 |
| 1.2 | 初始测试账号 | 种子数据或迁移中创建测试账号：用户名 **root**、密码 **root**；仅开发/测试环境启用或可配置关闭 |
| 1.3 | 验证码发送 | POST 请求发送验证码（入参：邮箱），系统发邮件验证码；防刷（同一邮箱频率限制） |
| 1.4 | 注册 | POST 注册（用户名、密码、邮箱、验证码），校验验证码通过后创建用户、写入 password_hash、标记 email_verified；生产环境启用密码复杂度校验（至少 8 位，含大小写和特殊字符），开发环境放宽 |
| 1.5 | 登录 | POST 登录（用户名或邮箱 + 密码），校验通过签发 JWT（access + refresh） |
| 1.6 | JWT 中间件 | 校验 access token、注入当前 user，保护需要登录的接口 |
| 1.7 | 刷新与登出 | refresh 换新 token；登出可黑名单或仅前端丢弃 token（按你安全策略） |
| 1.8 | 用户信息 API | GET 当前用户 profile（用户名、邮箱、状态等），供前端与配额展示 |

**依赖**：M0。  
**建议周期**：约 3–5 天。

---

### 模块二：套餐管理

**目标**：套餐与配额包配置、用户订阅、Stripe 支付与 Webhook、配额计算与校验，供任务模块扣减配额。

| 步骤 | 内容 | 验收标准 |
|------|------|----------|
| 2.1 | 套餐与配额包配置表 | subscription_plans、quota_package_plans（含 Stripe Price ID），可配置驱动 |
| 2.2 | 订阅与配额包模型 | subscriptions 表（user_id, plan_type, stripe_*, quota_per_month, quota_used, current_period_*）；quota_packages 表 |
| 2.3 | 注册赠送配额 | 用户注册成功后写入初始配额或首次「订阅」记录（5 个小视频一次性） |
| 2.4 | Stripe 订阅流程 | 创建 Checkout Session、成功回调 Webhook、创建/更新 subscription、更新配额 |
| 2.5 | 配额包购买 | Payment Intent / Checkout、Webhook、写入 quota_packages、增加用户可用配额 |
| 2.6 | Webhook 安全 | 验签、幂等、失败重试与日志 |
| 2.7 | 配额 API 与校验 | GET 用户当前配额/订阅信息；提供「扣减/退还配额」接口供任务模块调用；创建任务前校验可用配额 |

**依赖**：M0、用户管理。  
**建议周期**：约 1–1.5 周。

---

### 模块三：短视频任务

**目标**：单张图片生成单个视频的完整流程：视频服务抽象（Seedance、场景模板）、任务创建、异步生成、查询与下载。为长视频任务提供可复用的视频服务抽象与任务/下载能力。

| 步骤 | 内容 | 验收标准 |
|------|------|----------|
| 3.1 | 视频服务抽象 | Seedance 客户端封装（创建图生视频、查询状态、解析视频 URL）；模型与参数从配置读；分辨率/比例/时长（2–10 秒）与 API 参数映射 |
| 3.2 | 场景模板 | scene_templates 表 + 列表接口（供前端选择）；后台 CRUD 可放在管理后台模块 |
| 3.3 | 视频任务模型 | video_tasks 表（user_id, task_type=short/single, status, images JSONB, 分辨率/比例/logo_url, volcengine_task_ids, video_url, expires_at 等），迁移 |
| 3.4 | 图片上传 API | 上传图片到存储层，返回 URL 或 key；校验类型、大小 |
| 3.5 | 创建短视频任务 API | 入参：单张图片 key、场景模板 ID、分辨率、比例、时长、可选 logo；校验配额后创建 video_task、扣减配额、入队 Celery |
| 3.6 | 单视频 Celery 任务 | 调用视频服务创建图生视频、轮询状态、下载到存储、更新 video_tasks.video_url 与状态；失败时退还配额 |
| 3.7 | 任务查询与列表 API | 单任务详情、用户任务列表（分页、按状态筛选、按时间排序），支持 task_type 筛选 |
| 3.8 | 下载 API | 通过存储抽象生成访问 URL（过期时间按套餐 storage_days），仅允许任务所属用户下载 |

**依赖**：M0、用户管理、套餐管理。  
**建议周期**：约 1.5–2 周。

---

### 模块四：长视频任务

**目标**：多张图片分别生成视频片段后合并为一条长视频；支持转场与水印。复用模块三的视频服务抽象、video_tasks 表结构、任务列表与下载 API，仅扩展 task_type 与合并逻辑。

| 步骤 | 内容 | 验收标准 |
|------|------|----------|
| 4.1 | 创建长视频任务 API | 入参：多张图片 key（2–10 张）、统一场景模板 ID（或每图独立模板，按套餐）、统一时长、可选 logo；校验配额（按图片数扣减）、创建 video_task（task_type=merge）、入队合并任务 |
| 4.2 | 合并视频 Celery 任务 | 为每张图调用视频服务创建子任务并轮询；全部成功后下载片段到临时目录、moviepy 合并（转场淡入淡出）、可选 logo 水印；上传最终视频到存储、更新 video_tasks.video_url 与状态；任一步失败则退还配额 |
| 4.3 | 任务列表与下载 | 复用模块三的任务列表与下载 API，支持按 task_type 筛选「短视频/长视频」；长视频任务使用同一 video_url、expires_at 逻辑 |

**依赖**：M0、用户管理、套餐管理、短视频任务（复用视频服务抽象、任务表、上传与下载 API）。  
**建议周期**：约 1–1.5 周。

## 四、每步开发时建议自检

- **配置**：无 API Key、路径等硬编码；敏感项仅环境变量。
- **数据**：新表/字段都有迁移；关键操作有日志。
- **错误**：第三方调用有重试/超时；失败有明确错误码或文案。
- **安全**：上传校验类型与大小；接口有鉴权；支付 Webhook 验签。
- **运行与测试**：如果修改需要重启前后端才能验证，由我自动重启；端口被旧进程占用时，先停旧进程再按既定端口重启，不靠不断换端口规避。

## 六、对象存储策略（约定）

- **短期**：在服务器本地目录存储上传的图片、生成的视频和 Logo 等文件；配置项如 `STORAGE_TYPE=local`、`STORAGE_LOCAL_ROOT=/data/listinglive`。
- **接口**：所有读写通过统一「存储服务接口」进行（如：`save(key, stream, content_type)`、`get_url(key, expires_in)`、`delete(key)`），业务代码不直接依赖本地路径或 S3 SDK。
- **后续**：新增 S3 实现（如 `STORAGE_TYPE=s3`），实现同一接口并配置 bucket、region、CDN 等即可切换，无需修改任务、下载、清理等业务逻辑。

若需要，下一步可以从 **M0** 或 **模块一（用户管理）** 开始，按当前仓库结构给出具体目录和示例代码。
