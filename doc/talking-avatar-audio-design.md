# 对口型头像：根据图片自动生成解说词/音频 与 分段对齐

## 两个重点的先后关系

1. **重点一：根据图片自动生成解说词并转为音频**（按片段、与视频长度搭配好）
2. **重点二：用该音频 + 用户头像 生成对口型小头像视频**，再与当段房源视频合成

下文先说明「文案/音频自动生成」的含义和实现思路，再说明如何与每段视频长度对齐，最后回到口型头像的衔接方式。

---

## 一、文案/音频“自动生成”怎么理解？

这里的「自动」指：**用户不手写每段解说词，由系统根据「这张图 + 这段是什么场景」生成一句解说文案，再交给 TTS 变成音频**。

### 1.1 输入从哪来（和现有数据结构一致）

- **长视频**：每个片段（[LongVideoSegment](backend/models/long_video_segment.py)）已有  
  `image_key`、`scene_template_id`、`duration_seconds`。  
  场景模板在 [scene_templates.json](config/scene_templates.json) 里，有 `template_key`（如 `kitchen_pan`）、`name`（如「厨房亮点展示」），可推知「这一段是厨房/客厅/卧室…」。
- **短视频**：单张图 + 一个 scene_template，同样有「这张图 + 场景类型」。

所以「根据图片」= 用**当前片段的图片**；「自动」= 用**当前片段的场景类型**（+ 可选：图片内容）生成该段专属的解说词。

### 1.2 自动生成文案的几种做法

| 方式 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| **A. 纯模板** | 按 scene 类型固定一句，如厨房→「这是厨房，空间明亮整洁。」 | 实现简单、时长好控 | 与具体图片无关，千篇一律 |
| **B. 图+场景 → VLM 生成一句** | 把该片段图片 + 场景类型（如「厨房」）发给视觉语言模型，要求生成一句**房源解说**（中文/英文、限定字数），如「开放式厨房，岛台与不锈钢厨电齐全。」 | 和画面内容匹配、更自然 | 需调用 VLM、要控字数以控 TTS 时长 |
| **C. 模板 + VLM 补一句** | 模板给结构，如「这是厨房。{detail}。」；VLM 只生成 {detail}（一句短描述）。再拼成完整句做 TTS。 | 兼顾稳定与个性化 | 实现略复杂 |

**推荐**：先做 **B**（图+场景 → 一句解说），便于和「每段时长」约束一起控（见下）；若需更强可控性，再考虑 C。

### 1.3 技术链路（重点一）

```
对每个片段（如第 2 段厨房）：
  输入：该片段图片 image_key + 该片段 scene_template（得到场景类型 + duration_seconds）
    ↓
  步骤 1 - 生成文案
    - 调用 VLM（多模态大模型）：输入图片 + 提示语
    - 提示语示例：「这是房源视频的一段，场景为【厨房】。请用一句中文/英文解说词描述画面，适合配音，不超过 20 字。」
    - 输出：该片段专属的一句解说词 script_segment_2
    ↓
  步骤 2 - 文案 → 音频
    - TTS API：输入 script_segment_2，可选指定语速/音色
    - 输出：该片段的音频文件 audio_segment_2（时长 ≈ TTS 实际时长）
```

这样，「厨房这一段」的文案和音频就**只来自这一段的图和场景**，不会串到其他段。

## 二、如何与视频长度搭配？——「厨房的音频只在这一段」

目标：**每个片段里，解说只讲本段内容，且口型/音频时长不要超过该片段视频长度**。

### 2.1 按片段一一对应

- 长视频 = 多段（segment 1, 2, 3…），每段已有：
  - `duration_seconds`（如 5 秒）
  - 一张图、一个 scene_template（如厨房）
- **约定**：  
  - **片段 2 的图 + 场景** → 只生成**片段 2 的文案** → 只生成**片段 2 的 TTS 音频** → 只生成**片段 2 的口型小头像**。  
  - 合成时：片段 2 的房源视频（5 秒）与片段 2 的口型头像（≤5 秒）对齐，画中画或同轨剪辑。

这样「厨房的音频就只能在这一段」在数据流上自然满足：**按 segment 维度生成文案和音频，不做跨段混用**。

### 2.2 控制解说时长不超片段长度

- 每个片段有固定 `duration_seconds`（如 5 秒）。
- 希望：TTS 出来的音频时长 ≤ 该片段时长，这样口型小头像长度也不会超。

做法：

1. **在 VLM 生成文案时约束字数**  
   - 例如：根据 `duration_seconds` 估算「大约可读 X 字」（如 5 秒约 15–20 字中文），在提示语里写「不超过 X 字」。
2. **TTS 后校验**  
   - 若 TTS 实际时长 > 该片段 `duration_seconds`，可：  
     - 截断音频到 `duration_seconds`，或  
     - 重新生成更短文案再 TTS（重试一次）。

这样既保证「厨房的解说只在厨房这一段」，又保证「这段解说/口型不会拖到下一段」。

### 2.3 短视频（单段）

- 只有一段，`duration_seconds` 已知；同样：**这张图 + 该段 scene_template** → 生成一句解说 → TTS → 口型小头像，时长 ≤ 该段时长即可。

## 三、多段文案合在一起的割裂感怎么处理？

若每段**独立**按「图 + 场景 → 一句」生成，再按顺序播放，容易出现：

- 每句都是「这是客厅。」「这是厨房。」「这是卧室。」，**句式雷同、像报菜名**；
- 段与段之间**没有过渡**，听起来像多段独立配音硬接在一起；
- 语气、详略不统一（有的很细、有的过简）。

可以从「生成时带上下文」或「生成后做一次润色」两方面缓解，按实现成本从低到高简述如下。

### 3.1 生成时带「上文」：续写式提示

- 生成**第 N 段**文案时，除了「本段图片 + 本段场景」，再给模型**前一段（或前几段）的解说词**。
- 提示语示例：「上一段解说词是：『客厅开阔，采光很好。』本段是【厨房】的镜头（见附图）。请写一句本段解说，要自然衔接上一句，不要再用『这是厨房』开头，可改为『再来看厨房』或直接描述画面。」
- 这样从第二段开始就有「接续感」，避免每段都「这是 XXX」。代价是：生成顺序必须从第 1 段到第 N 段串行，且每段调用都带一点上文 token。

### 3.2 生成后做一次「连贯性润色」（推荐）

- **第一遍**：仍按段独立生成（可并行），得到 draft_1, draft_2, …, draft_N。
- **第二遍**：只做**纯文本**处理，不再看图。把 [draft_1, draft_2, …, draft_N] 拼成一篇，发给**只吃文本的 LLM**，提示语示例：「以下是房源视频各段解说词，按顺序对应客厅、厨房、卧室…。请润色：保持每段仍是一句、字数不要明显增加，但让语气连贯、段与段有过渡、避免每句都是『这是…』开头。直接输出润色后的各段文案，每段一行。」
- 输出再按段拆开，分别做 TTS。这样**每段内容仍与对应画面匹配**（因为第一遍是按图生成的），只是用第二遍做「衔接与语气统一」，成本多一次纯文本 LLM 调用，通常比多段 VLM 便宜很多。

### 3.3 固定「过渡句」模板 + 每段只生成内容句

- 事先按**段落顺序**定好过渡语，例如：第 1 段无前缀，第 2 段前缀「接下来是厨房，」，第 3 段前缀「卧室部分，」，等等。
- 每段 VLM 只生成**内容部分**（如「岛台与厨电齐全，空间明亮」），再和模板拼成完整句：「接下来是厨房，岛台与厨电齐全，空间明亮。」
- 优点：实现简单、节奏可控、割裂感小。缺点：过渡句较固定，变化少。

### 3.4 小结

| 做法 | 效果 | 成本 / 复杂度 |
|------|------|----------------|
| 生成时带「上一段」文案（3.1） | 有接续感，减少「这是…」重复 | 需串行生成，每段多带一点上文 token |
| 生成后纯文本润色一遍（3.2） | 连贯、过渡自然，且保留每段与画面的对应 | 多 1 次文本 LLM 调用，易落地 |
| 固定过渡模板 + 每段只生成内容（3.3） | 稳定、不割裂 | 过渡句选择少 |

**推荐**：先做 **3.2**（两遍：按图生成 → 再整体润色），在不大改现有「按段生成」管线的前提下减轻割裂感；若希望再稳一点，可结合 **3.3** 给第二段起加简单过渡模板，再由润色统一语气。

## 四、两个重点在整体管线中的位置

```text
长视频示例（片段 1=客厅，片段 2=厨房，片段 3=卧室）：

  片段 1:
    图1 + 场景「客厅」 + duration_seconds=5
      → [重点一] VLM 生成客厅解说词 → TTS → audio_1
      → [重点二] audio_1 + 用户头像 → 口型小头像视频 clip_1
      → 合成：片段 1 房源视频 + clip_1（画中画或前后衔接）

  片段 2:
    图2 + 场景「厨房」 + duration_seconds=5
      → [重点一] VLM 生成厨房解说词 → TTS → audio_2
      → [重点二] audio_2 + 用户头像 → 口型小头像视频 clip_2
      → 合成：片段 2 房源视频 + clip_2

  片段 3: 同理…
```

- **重点一**：根据**图片 + 场景**自动生成**该段**解说词，并转为**该段**音频（且时长 ≤ 该段 `duration_seconds`）。
- **重点二**：用**该段音频 + 用户头像**生成**该段**对口型小头像，再与**该段**房源视频合成。

这样既实现了「根据图片自动生成音频」，又实现了「厨房的音频只在这一段」和「与视频长度搭配好」，并把口型头像放在明确第二步。

## 五、VLM 成本参考（选读）

用于「图 + 场景 → 一句解说」的视觉语言模型，按**每张图 + 短文本 prompt + 一句输出**估算，大致量级如下（均为公开 API、按 token 或按量计费，具体以各厂商最新价格为准）：

| 厂商 / 模型 | 计费方式 | 单次调用量级（1 图 + 短 prompt + 1 句输出） | 备注 |
|-------------|----------|---------------------------------------------|------|
| **OpenAI GPT-4o** | 输入/输出按 token，图按分辨率折算 token | 约 $0.01–0.03/次（视分辨率） | 效果稳定，英文/中文均可 |
| **OpenAI GPT-4o mini** | 同上 | 理论上更低，但社区反馈 vision 的 image token 消耗偏大 | 需实测单次成本 |
| **Anthropic Claude** | 按 token，图：tokens ≈ (宽×高)/750 | 约 $0.001–0.005/次（如 1000×1000 约 $0.004） | 文档明确，易估算 |
| **Google Gemini 1.5 Flash** | 按 token，图/文统一 | 约 $0.0005–0.002/次（输入 $0.075/百万 token） | 价格较低，适合大批量 |
| **Google Gemini 1.5 Flash-8B** | 同上 | 约一半于上 | 更便宜，限流更高 |
| **火山引擎 / 豆包** | 按 token 或按次套餐 | 需查当前多模态/视觉模型报价（有按千次、按 token 多种） | 与现有 seedance 同厂，账号与合规统一 |

**和业务的关系**：每个长视频若 5 段，则每成片约 5 次 VLM 调用；短视频 1 次。若日产 100 个长视频 ≈ 500 次/天，按 Gemini Flash 量级约几美元/天级别，可按实际用量选模型与厂商。若希望与现有视频管线同一供应商，可优先查火山引擎多模态/视觉 API 的当前定价与配额。

## 六、实现时需要的扩展（概要）

- **数据**：每个片段是否启用「对口型」、是否使用自动文案（可扩展 LongVideoSegment 或任务级配置）。
- **服务**：  
  - 文案生成：调用现有或新增大模型（VLM）接口，输入图片 + 场景类型 + 字数/时长约束。  
  - TTS：接入一家 TTS API，输入文案、输出音频文件。  
  - 口型头像：接入 D-ID/HeyGen 等，输入头像图 + 该段音频，输出该段 talking head 视频。  
- **管线**：在现有「按片段生成房源视频」之后，对需要对口型的片段依次做「生成文案 → TTS → 口型视频 → 与该段视频合成」，再与 logo/名片等后处理一起输出成片。

以上是对「根据图片自动生成解说词/音频」的含义、与视频长度和分段的搭配方式、以及两个重点（先生成音频、再生成口型小头像）的整理。
